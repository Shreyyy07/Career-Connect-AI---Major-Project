1.  Purpose & Scope
This document defines the complete product requirements for the HR Recruiter Portal of Career Connect AI. It specifies every feature, backend flow, API interaction, email verification mechanism, dashboard component, and business rule that governs the HR recruiter experience — from account registration through job description management, candidate review, anti-cheat monitoring, and final hiring decision.

This PRD is a companion document to the Candidate Portal PRD. Together they constitute the full platform specification. This HR PRD is written to be fully implementation-ready.

1.1  HR Recruiter Definition
An HR User is a human resources professional or hiring manager who creates an account on behalf of their company. They:
Register with First Name, Email, Password, and Company Name (four mandatory fields).
Verify their email address before accessing any feature.
Upload, manage, and publish Job Descriptions (JDs) that candidates can browse and interview for.
View all candidates who have completed AI interviews for their JDs, including full evaluation reports, anti-cheat logs, and multimodal scores.
Make shortlisting decisions (Shortlist / Reject / Keep Under Review) on individual candidates.
Access aggregated HR analytics dashboards.

1.2  What HR Users Cannot Do
Upload or view candidate resumes directly (they see extracted skills summaries only — full resume file is not accessible to preserve candidate privacy).
Modify or override AI-generated scores.
View candidates who applied to jobs posted by other HR users.
Access the Candidate Portal flows (resume upload, JD selection for personal matching, AI interview as a candidate).
Access admin system configuration panels.

2.3  HR Login Flow
Post-verification, HR users log in with Email + Password. The login screen is shared with candidates but routes to the HR-specific dashboard based on the user.role value in the JWT.

Step
Backend Logic
1. Submit login
POST /api/auth/login — Body: { email, password }
2. Email lookup
Query users table. If not found → 401 "Invalid credentials". (Same message for both not-found and wrong-password — prevents email enumeration attacks.)
3. Verification check
If status == "pending_verification" → 403 with body { error: "EMAIL_NOT_VERIFIED", message: "Please verify your email before logging in." }. Frontend shows a specific screen with resend link.
4. Password check
bcrypt.verify(password, stored_hash). Failure → 401.
5. JWT issuance
JWT payload: { user_id, email, role: "hr", name, company_name, iat, exp: +24h }. Return 200 { token, user: { id, name, email, role, company_name } }.
6. Dashboard routing
Frontend checks user.role from JWT payload. If "hr" → route to /hr/dashboard. If "candidate" → route to /candidate/dashboard. This prevents an HR user from accidentally accessing the candidate flow.

4.  Job Description Management
4.1  JD Creation Rules
Only users with role = "hr" can create, edit, or delete Job Descriptions. This is enforced at the API level (middleware role check returns 403 if role != "hr").
A JD can be in one of three states: Draft (visible only to the creating HR user), Active (visible to all candidates for browsing and interviewing), Closed (hidden from candidates, preserved for HR analytics).
HR users can create unlimited JDs under their account.

Step
1
Auth Check
Middleware verifies JWT. Checks role == "hr". If not → 403 Forbidden. If yes → proceed.

POST /api/jobs → 201 { jd_id, title, status, created_at }.

6.  Anti-Cheat Monitoring — HR View
The anti-cheat system runs continuously during every candidate interview session. HR users see the results on their dashboard in a dedicated Anti-Cheat Panel. Candidates are informed that monitoring is active (shown in the pre-interview consent screen) but cannot see the specific events flagged.

6.1  Anti-Cheat Detection Modules
Module
Technology
Detects
Trigger Threshold
Face Verification
DeepFace
Identity mismatch between the initial face captured at session start and the face appearing throughout the interview.
Confidence match < 90% triggers WARNING. < 70% triggers CRITICAL flag.
Face-Out Detection
DeepFace + OpenCV
Candidate's face is not visible in the camera frame.
Face absent for > 5 continuous seconds triggers WARNING. > 15 seconds triggers CRITICAL.
Multiple Person Detection
YOLOv8 / YOLOv11
More than one person detected in the interview frame.
Any detection of 2+ persons triggers immediate CRITICAL flag. Session flagged for review.
Liveness / Fake Video
Haarcascade liveness check
Pre-recorded video playback instead of live camera feed.
Liveness probability < 60% triggers WARNING. Sustained < 40% → CRITICAL.
Screen Switching
Browser Page Visibility API
Candidate switches to another browser tab or application during the interview.
Any tab switch > 10 seconds logged as WARNING. Repeated switches flagged as CRITICAL.

6.2  Event Severity Levels
Severity
Label
Action Taken During Interview
HR View
LOW
Informational
Logged. No interruption to candidate.
Shown in expanded log. Does not affect final score.
WARNING
Suspicious Activity
On-screen warning shown to candidate: "Maintain proper interview posture." Interview continues.
Yellow badge in candidate row. Count shown in summary.
CRITICAL

6.3  Anti-Cheat Panel on HR Dashboard
The Anti-Cheat Panel shows a unified log across all candidates. HR can:
View all events chronologically, or filter by candidate, session, severity, or event type.
See per-candidate anti-cheat score: 100 = fully clean session. Deducted based on severity and frequency of events.
See flagged sessions prominently highlighted at the top of the candidate list.
Toggle an "Integrity Risk" filter on the candidate list to show only candidates with CRITICAL events.

6.4  Anti-Cheat Data Schema
Table: anticheat_events — { event_id, session_id, candidate_id, timestamp, event_type (enum), severity (enum: LOW/WARNING/CRITICAL), frame_snapshot_url (optional), details_json, created_at }.
Aggregated per session: anticheat_summaries — { summary_id, session_id, warning_count, critical_count, overall_integrity_score (0–100), is_flagged (boolean) }.
is_flagged = true if critical_count ≥ 2 OR overall_integrity_score < 60.

11.  Security & Privacy — HR Module
Concern
Implementation
HR-to-candidate data access
Every /api/hr/* endpoint checks: (a) JWT is valid. (b) user.role == "hr". (c) For evaluation-specific endpoints, the job_id in the evaluation matches a JD created by this HR user (ownership check). Violation → 403.
Cross-HR isolation
HR User A cannot see HR User B's candidates, JDs, or analytics. All queries are scoped by hr_user_id.
Candidate resume files
HR users CANNOT download the original resume PDF/DOCX. They see only the extracted, structured skills data. Raw resume files are stored in private S3 with pre-signed URLs accessible only to the owning candidate.
Anti-cheat snapshots
Video frame snapshots captured during anti-cheat events are stored in private S3 with 30-day retention. Accessible only to HR for review. Not included in the candidate-facing report PDF.
Rate limiting
All /api/hr/* endpoints: 200 requests/minute per authenticated HR user. Throttle with 429 response. Prevents scraping of candidate data.
GDPR / data deletion
Candidate can request data deletion via account settings. Cascade deletes: resume file (S3), parsed data, embeddings, emotion logs, speech features. Evaluation records are anonymised (candidate_id set to NULL) and retained for 90 days for HR analytics before full deletion.
