1.  Purpose & Scope
This document defines the complete product requirements for the Candidate Portal of Career Connect AI. It specifies every feature, backend data flow, API interaction, email verification mechanism, UI screen, and business rule that governs the candidate experience — from initial account registration through AI interview completion and final report review.

This PRD is written to be implementation-ready: a developer can read this document and build the candidate module end-to-end without requiring additional verbal clarification.

1.1  Candidate Definition
A Candidate is any job-seeker who creates an account on Career Connect AI. They:
Register with First Name, Email, and Password (three mandatory fields).
Verify their email address using a one-time link sent by the platform before accessing any feature.
Upload their own resume (PDF or DOCX). They cannot upload or create Job Descriptions.
Browse and select Job Descriptions posted by HR users.
Initiate an AI-powered interview for a selected JD.
View their skill gap analysis, interview performance scores, and evaluation reports on their personal dashboard.

1.2  What Candidates Cannot Do
Upload, create, edit, or delete Job Descriptions.
View other candidates' profiles, resumes, or interview reports.
Access the HR dashboard or HR analytics.
Change evaluation scores or report contents.
Access admin configuration panels.

Email verification uses a one-time signed link mechanism. This is implemented at zero cost using one of the following free-tier SMTP/transactional email services:


2.2.1  Verification Token Generation
When a candidate submits the registration form successfully:
Backend creates a user record in the database with status = "pending_verification".
Backend generates a cryptographically secure random token: 64-character hex string using secrets.token_hex(32) in Python.
Token is stored in the verification_tokens table with columns: token (hashed with SHA-256), user_id (FK), created_at, expires_at (= created_at + 24 hours), used (boolean, default False).
Backend sends a verification email to the registered address using the chosen free SMTP service.
Candidate is shown a screen: "We sent a verification link to your-email@example.com. Please check your inbox."

2.2.2  Verification Email Template

Verification link format:
  https://[domain]/auth/verify?token=[64-char-hex-token]

2.2.3  Verification Click Flow (Backend)
Candidate clicks the verification link in their email.
Frontend extracts token from the URL query parameter and calls: GET /api/auth/verify?token={token}
Backend: Hash the received token using SHA-256. Look up the hashed value in verification_tokens. If found: check used == False AND expires_at > NOW(). If valid: set users.status = "active", set verification_tokens.used = True. Return 200 with success message.
If token is expired: return 410. If token already used: return 409. If token not found: return 404.
Frontend redirects verified candidate to the Login page with a success toast: "Email verified! You can now sign in."

2.2.4  Resend Verification Email
If the candidate does not receive the email or it expires, they can request a new link on the "Check your email" screen:
Candidate clicks "Resend verification email".
Frontend calls: POST /api/auth/resend-verification with body { email: "candidate@example.com" }.
Backend: Validate that the email exists and status == "pending_verification". Invalidate any existing unused tokens for this user. Generate a new token. Send a new email. Return 200.
Rate limit: Maximum 3 resend requests per email per hour to prevent abuse.

2.3  Login Flow
Post-verification, a candidate logs in using Email + Password. The login screen has exactly two fields.


2.4  Password Reset Flow
"Forgot Password" link on the login page triggers: POST /api/auth/forgot-password with { email }.
Backend generates a 6-digit OTP (not a link, to differentiate from the verification link). OTP valid for 10 minutes.
OTP emailed to the candidate. Frontend shows an OTP input screen.
Candidate submits: POST /api/auth/reset-password with { email, otp, new_password }.
Backend validates OTP, expiry, and updates the bcrypt hash. Invalidates the OTP. Returns 200.

4.  Resume Upload & Parsing
4.1  Upload Rules
Accepted formats: PDF and DOCX only. Any other format is rejected with a specific error.
Maximum file size: 5 MB. Files larger than 5 MB are rejected before uploading.
A candidate may upload a new resume at any time. The new resume replaces the old one. All future match computations use the latest resume.
The candidate cannot upload a Job Description — this action is locked behind the HR role at the API level.

6.  Skill Gap Analysis & Training Recommendations
6.1  Gap Analysis Panel on Dashboard
The skill gap panel appears automatically after a match computation. It is persistent — the candidate can return to the dashboard and view the gap analysis for the last matched JD without re-triggering the match.


6.2  Recommendation Card Structure
For each Missing Skill, the platform renders a recommendation card containing:
Skill name (bold, coloured heading).
Why you need this: 1–2 sentence explanation of its relevance to the selected role.
Recommended Learning Path: Course/book/tutorial name, platform (Coursera/Udemy/etc.), estimated hours.
Priority level: High (core JD requirement) / Medium / Low.
Completion tracker: Candidate can mark each recommendation as "Not Started", "In Progress", or "Done". This state is persisted to the DB and shown as a progress indicator on the dashboard.

6.3  Recommendation Storage & Progress Tracking
Table: skill_recommendations — { rec_id, match_id, candidate_id, skill_name, reason, resource_name, resource_url, platform, est_hours, priority, status (enum: not_started / in_progress / done), updated_at }.
API: PATCH /api/recommendations/{rec_id} — Body: { status: "in_progress" }. Only the owning candidate can update their own recommendation status.
Dashboard progress bar shows: "X of Y recommended skills completed."

7.3  Question Generation Rules
Total questions per session: 5 to 8 (configurable, default 7).
Question types: Technical (role-specific), Behavioural (situational), Communication-quality (open-ended).
No question is repeated within a session. Questions are generated dynamically by Gemini using the RAG context — they are NOT from a static template bank.
RAG prompt instructs Gemini: "Generate interview questions for a [job_title] role. Consider this job description: [jd_text]. The candidate has [experience_years] years of experience and these skills from their resume: [skills_list]. Generate role-specific, non-repetitive questions that test technical depth and communication ability."

8.3  Semantic Answer Evaluation (Gemini LLM)
For each question-answer pair, Gemini evaluates: (a) Factual correctness — Is the answer technically accurate for the role? (b) Depth — Does the candidate go beyond surface-level? (c) JD alignment — Does the answer reference skills and responsibilities in the JD? (d) Completeness — Are all parts of the question addressed?
Output: { semantic_score: 0–100, jd_relevance_score: 0–1, evaluation_notes: "string" }.
Scores across all question-answer pairs are averaged to form the session-level Semantic Score.

9.1  Candidate Score Visibility
Candidates see all four component scores + the final composite score on their Evaluation Report Card.
Score thresholds shown as colour bands: ≥ 80 = Excellent (Green), 60–79 = Good (Teal), 40–59 = Average (Amber), < 40 = Needs Work (Red).
Candidates do NOT see the HR-set shortlist decision (Shortlisted / Rejected) directly. They see their scores and Gemini-generated qualitative feedback.


10.  Evaluation Report — Candidate View
10.1  Report Card (On-Screen)
Immediately after an interview ends, the candidate is shown an "Interview Card" screen containing:
Final composite score (large, colour-coded score ring animation).
All four component scores displayed as smaller score rings: Semantic, JD Match, Emotion Stability, Communication.
Emotion timeline: A horizontal chart showing the candidate's emotional state distribution across the interview duration.
Gemini-generated Strengths (2–3 bullet points).
Gemini-generated Areas for Improvement (2–3 bullet points).
Role Alignment Summary: A 2–3 sentence narrative generated by Gemini explaining how well the candidate fits the selected JD.
A "Download Full Report" button to get the PDF version.
